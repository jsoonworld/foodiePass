# FoodiePass Frontend - Hypotheses

> **목적**: 프론트엔드 구현 시 "왜 이 UI를 만드는가"를 이해하기 위한 핵심 가설 요약

---

## 핵심 문제

**여행객이 외국어 메뉴판을 마주할 때 '정보 비대칭'으로 인해 '주문 실패의 불안'을 겪습니다.**

---

## 검증할 핵심 가설

### H1: 핵심 가치 가설
> "[텍스트 번역만]으로는 여전히 불안하지만, [사진/설명/환율]이 포함된 시각적 메뉴가 제공될 경우, '주문에 대한 확신'을 갖게 된다."

**프론트엔드의 역할**:
- Control UI: 텍스트 + 환율만 표시
- Treatment UI: 사진 + 설명 + 텍스트 + 환율 표시
- **두 UI의 차이만으로 가설 검증**

---

### H3: 사용자 행동/인지 가설
> "[시각적 메뉴] 사용 집단은 [텍스트 번역만] 집단 대비, '확신을 갖고 주문할 수 있다' 응답률이 **2배 이상 높을 것**."

**프론트엔드의 역할**:
- 설문 UI: "이 정보만으로 확신을 갖고 주문할 수 있습니까? (Yes/No)"
- abGroup에 따라 다른 UI 렌더링 (Control vs Treatment)

---

## 프론트엔드 기능 매핑

| 가설 | 프론트엔드 기능 | 컴포넌트 |
|---|---|---|
| **H1** | Control UI (텍스트 전용) | `TextOnlyMenu` |
| **H1** | Treatment UI (시각적 메뉴) | `VisualMenu` |
| **H3** | 설문 UI | `SurveyModal` |
| **공통** | 업로더 UI | `MenuUploader` |

---

## MVP 범위

### ✅ 구현 필요
- **F-01**: 웹 업로더 (이미지 + 언어/화폐 선택)
- **F-06**: 시각적 메뉴 UI (Treatment 그룹)
- **F-07**: 텍스트 전용 UI (Control 그룹)
- **F-09**: 확신도 설문 UI

### ❌ 제외
- 로그인/회원 기능
- 장바구니
- 메뉴 저장/히스토리

---

## 핵심 원칙

### 1. A/B 테스트 무결성
- 백엔드가 제공한 `abGroup`에 따라 **정확히** Control 또는 Treatment UI 렌더링
- 사용자가 그룹 변경 불가

### 2. 단순성 우선
- MVP는 복잡한 기능 불필요
- 업로드 → 결과 표시 → 설문 제출 플로우만 구현

### 3. 모바일 우선
- 여행객은 주로 모바일 사용
- 반응형 디자인 필수
